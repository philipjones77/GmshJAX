# Status Standard

## Purpose

This document defines what belongs in `docs/common/status/`.

## Placement Rule

Put a document in `docs/common/status/` when the main question is:

- where is the shared track today
- what remains to be done in the shared layer
- what cross-cutting gaps or plans are active

## Writing Rule

Status documents should:

- describe current state, gaps, or plans
- distinguish done, in progress, and planned work where that helps
- stay focused on shared-track concerns

Status documents should avoid:

- restating full implementation details better kept in `implementation/`
- mixing backend-specific TODOs into common-track status unless the issue is genuinely cross-cutting
